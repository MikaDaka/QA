# SDLC, STLC, виды тестирования и жизненный цикл бага
## SDLC — жизненный цикл разработки ПО
SDLC описывает путь продукта от идеи до поддержки. Всё начинается с анализа требований: команда собирает информацию о том, что нужно пользователю, какие задачи должен решать продукт и какие ограничения существуют. На этом этапе тестировщик изучает требования, ищет противоречия и помогает сделать их понятными и проверяемыми.

После анализа следует проектирование. Разработчики продумывают архитектуру, выбирают технологии, создают схемы взаимодействия. Тестировщик в это время формирует стратегию тестирования, оценивает риски и определяет подходы.

Когда начинается разработка, тестировщик готовит тестовую документацию и тестовые данные. Как только появляется рабочая версия продукта, начинается тестирование: проверка функционала, поиск дефектов, регрессия. После успешной проверки продукт выкатывается пользователям, а тестировщик выполняет финальный smoke‑тест. На этапе поддержки исправляются ошибки и добавляются улучшения, а тестировщик проверяет исправления.

## STLC — жизненный цикл тестирования
STLC идёт параллельно SDLC и описывает процесс тестирования. Он начинается с анализа требований, затем переходит к планированию, где определяется объём тестирования, инструменты и подходы. Далее создаются тест-кейсы и чек-листы, подготавливается окружение, выполняются тесты и фиксируются баги. Завершается цикл анализом результатов и подготовкой отчёта о качестве продукта.

## Виды тестирования
### 1. По уровню тестирования
### Unit Testing
Проверяются отдельные функции или методы.
Что проверяется: корректность логики, обработка входных данных, возвращаемые значения.
Когда применяется: на ранних этапах разработки, при изменении конкретной функции.

### Integration Testing
Проверяется взаимодействие нескольких модулей.
Цели: выявить проблемы на стыках систем, проверить корректность API, убедиться в правильной передаче данных.
Примеры: взаимодействие фронтенда и бэкенда, работа API и базы данных.

### System Testing
Проверяется система целиком.
Что включает: функциональные и нефункциональные проверки, пользовательские сценарии.
Цель: убедиться, что продукт работает как единое целое.

### Acceptance Testing
Проверяется готовность продукта к релизу.
Виды: внутреннее и пользовательское (UAT).
Цель: подтвердить, что продукт решает задачи бизнеса.

## 2. По степени автоматизации
### Ручное тестирование
Плюсы: гибкость, подходит для UI/UX, легко начать.
Минусы: медленно, человеческий фактор, плохо масштабируется.

### Автоматизированное тестирование
Плюсы: быстрое выполнение, высокая точность, идеально для регрессии.
Минусы: требуется программирование, сложность поддержки, не подходит для UX.

## 3. По доступу к коду
### Black Box Testing
- тестировщик не знает внутреннюю структуру кода;
- работает с интерфейсом и документацией;
- подходит для UI, API, функционального тестирования.

### White Box Testing
- тестировщик знает структуру кода;
- используется разработчиками;
- включает покрытие кода и проверку логики.

### Gray Box Testing
- частичное знание внутренней структуры;
- тестировщик использует SQL, логи, архитектуру;
- сочетает преимущества двух подходов.

## 4. По цели тестирования
### Функциональное тестирование
Проверяет, что система делает то, что должна.
Включает: проверку кнопок, форм, логики, API, бизнес‑процессов.

### Нефункциональное тестирование
1. Нагрузочное тестирование
- проверка работы под нормальной и повышенной нагрузкой;
- измерение времени отклика.

2. Стресс‑тестирование
- проверка системы в экстремальных условиях;
- определение пределов устойчивости.

3. Тестирование производительности
- скорость работы;
- пропускная способность;
- стабильность под нагрузкой.

4. Тестирование безопасности
- поиск уязвимостей;
- проверка прав доступа;
- попытки SQL Injection, XSS.

5. Юзабилити‑тестирование
- удобство интерфейса;
- логичность навигации;
- понятность элементов.

6. Тестирование совместимости
- разные браузеры;
- разные устройства;
- разные ОС.

7. Тестирование локализации
- корректность перевода;
- форматы дат и валют;
- отображение текста.

## 5. По времени проведения
### Smoke Testing
- быстрая проверка, что приложение запускается;
- проверяются самые критичные функции.

### Sanity Testing
- проверка конкретной исправленной фичи;
- узконаправленная проверка.

### Regression Testing
- проверка, что новые изменения не сломали старый функционал;
- один из самых важных типов.

### Re-test
- повторная проверка конкретного бага после исправления.

### Exploratory Testing
- исследовательское тестирование без заранее написанных сценариев;
- помогает находить неожиданные баги.

## Жизненный цикл бага (Bug Life Cycle)
Жизненный цикл бага описывает путь ошибки от момента обнаружения до полного закрытия.

### Основные статусы
- New - баг создан тестировщиком;
информация собрана, но баг ещё не рассмотрен.

- Open - баг подтверждён как реальная проблема;
принят в работу менеджером или тимлидом.

- Assigned - баг назначен конкретному разработчику;
начинается анализ.

- In Progress - разработчик исправляет ошибку;
ведётся работа над кодом.

- Fixed - разработчик сообщает, что баг исправлен;
изменения готовы к проверке.

- Ready for Testing / To Verify - баг возвращён тестировщику;
требуется повторная проверка.

- Verified - тестировщик подтвердил, что баг исправлен;
функционал работает корректно.

- Closed - баг окончательно закрыт;
работа завершена.

### Альтернативные статусы
- Rejected / Not a Bug - описанное поведение не является ошибкой;
баг отклонён.

- Reopened - баг всё ещё существует после проверки;
снова открыт.

- Deferred / Postponed - исправление отложено;
баг не критичен или требует больших изменений.
